Runtime trace evidence:
- Claude represents permissions as modes plus sourced rules. External modes are `acceptEdits`, `bypassPermissions`, `default`, `dontAsk`, and `plan`; internal runtime modes can add `auto` and `bubble`; rules carry `allow`, `deny`, or `ask` behavior and sources such as user, project, local, flag, policy, CLI, command, and session (`permissions.ts:16-80`).
- Claude permission decisions are not just boolean. They can be allow, ask, deny, or passthrough, with typed reasons for rule, mode, tool/subcommand result, permission prompt tool, hook, async agent, sandbox override, classifier, working directory, safety check, or other (`permissions.ts:241-324`).
- Claude's pipeline checks deny rules, ask rules, tool-specific checks, user-interaction requirements, content-specific ask rules, and safety checks before bypass mode or always-allow rules can grant execution. Bypass does not bypass configured content-specific asks or safety checks (`permissions.ts:1060-1320`).
- Claude transforms ask to deny in `dontAsk`, applies auto mode after ordinary rule/tool checks, runs PermissionRequest hooks before headless auto-deny, tracks auto-mode denial counts, and aborts headless sessions when denial limits are exceeded (`permissions.ts:473-535`, `permissions.ts:929-1058`).
- Claude filesystem permissions have dedicated read/write checks. Reads and writes examine explicit deny/ask rules, working-directory scope, internal readable/editable paths, session `.claude` allowances, dangerous path safety, `acceptEdits`, always-allow rules, and suggestion generation (`filesystem.ts:1028-1465`).
- Claude persistence is explicit permission-update application. Updates can set mode, add/replace/remove rules, or add/remove directories in memory; only local, user, and project settings destinations are persistable, while policy/flag/command sources are not ordinary editable stores (`PermissionUpdate.ts:55-216`, `PermissionUpdate.ts:222-389`).
- Claude managed policy mode can force rule loading to policy settings only and disable/hide ordinary always-allow options; settings writes preserve unknown fields and dedupe normalized rules (`permissionsLoader.ts:27-44`, `permissionsLoader.ts:91-156`, `permissionsLoader.ts:229-296`).
- Claude auto mode strips dangerous classifier permissions on entry and restores them on exit, with auto-mode activation gated by feature/config/circuit-breaker logic and CLI/settings mode selection (`permissionSetup.ts:267-553`, `permissionSetup.ts:581-811`).
- Claude interactive approval is a race among local dialog callbacks, bridge callbacks, channel replies, PermissionRequest hooks, and classifier checks. `useCanUseTool` creates a permission context, resolves immediate allow/deny, queues a `ToolUseConfirm` for asks, and logs decisions; `interactiveHandler` owns resolve-once and relay cleanup (`useCanUseTool.tsx:27-182`, `PermissionContext.ts:96-190`, `interactiveHandler.ts:43-120`, `interactiveHandler.ts:300-430`).
- Codex represents baseline authority as a `PermissionProfile`: managed filesystem plus network, disabled sandbox, or external sandbox. It separately tracks `ActivePermissionProfile` identity for clients and `AdditionalPermissionProfile` overlays for per-command or granted turn/session permissions (`models.rs:264-315`, `models.rs:408-485`).
- Codex approval policy is source-typed as `AskForApproval`: `UnlessTrusted`, `OnFailure`, `OnRequest`, `Granular`, or `Never`. Granular approval has independent gates for sandbox approval, exec-policy rules, skill approval, `request_permissions`, and MCP elicitations (`protocol.rs:837-905`).
- Codex model-visible permissions are injected as developer-context instructions generated by `PermissionsInstructions`, describing sandbox mode, approval policy, writable roots, denied reads, approved command prefixes, and the `request_permissions` tool when enabled. This is prompt guidance, not the enforcement layer (`permissions_instructions.rs:49-312`).
- Codex enforcement flows through `ExecApprovalRequirement`: skip, needs approval, or forbidden. The default requirement depends on `AskForApproval` plus filesystem sandbox kind, and granular policy can convert a would-be prompt into `Forbidden` (`sandboxing.rs:158-238`).
- Codex orchestration asks for approval before first execution when required, routes to guardian when configured or strict auto review is active, rejects non-approved decisions, and then chooses the initial sandbox. Denied-read restrictions prevent unsafe no-sandbox escalation because denied reads exist only inside the sandbox (`orchestrator.rs:132-235`, `orchestrator.rs:551-579`, `sandboxing.rs:246-303`).
- Codex shell and unified-exec approvals cache by canonicalized command, cwd, TTY where applicable, sandbox permissions, and additional permissions. Apply-patch approvals cache per environment and file path, because session approval must be reusable for subsets of patch targets (`shell.rs:94-206`, `unified_exec.rs:82-222`, `apply_patch.rs:40-179`).
- Codex approval requests carry proposed exec-policy amendments, proposed network-policy amendments, additional permission overlays, available decisions, command/cwd/reason, and approval IDs. Review decisions can approve once, approve for session, approve with exec-policy amendment, apply a network-policy amendment, deny, time out, or abort (`approvals.rs:217-321`, `protocol.rs:3781-3815`).
- Codex `request_permissions` is model-callable only when the feature adds the handler. The handler resolves the turn environment, parses/normalizes requested additional permissions relative to cwd, rejects empty requests, calls `request_permissions_for_environment`, and returns the approved response JSON to the model (`spec_plan.rs:644-662`, `request_permissions.rs:20-120`).
- Codex app-server fans core events into client approval requests for file changes, command execution, and permission requests; TUI normalizes approval request params and derives available decisions based on network context, additional permissions, or exec-policy amendments (`common.rs:1422-1454`, `bespoke_event_handling.rs:511-650`, `bespoke_event_handling.rs:765-800`, `approval_events.rs:1-119`).

What the model sees:
- Claude's model sees tool schemas and ordinary tool results, but not the raw permission dialog internals as a separate model-facing policy spec in this mechanism. Permission results can feed back as allow/deny/ask behavior, updated input, denial messages, hook feedback, or content blocks produced by the permission context (`PermissionContext.ts:264-345`).
- Claude's headless PermissionRequest hooks can update input or permissions before the model receives a denial/result path; channel approval replies are intercepted before becoming chat turns, so a remote "yes" response does not enter the conversation (`permissions.ts:929-952`, `interactiveHandler.ts:300-315`).
- Codex's model sees generated developer-context permission instructions and, when enabled, the `request_permissions` tool contract. The prompt can tell the model when escalation is allowed, when denied reads are not requestable, what command prefixes are already approved, and how granular approval behaves (`permissions_instructions.rs:113-312`).
- Codex's model sees the serialized `RequestPermissionsResponse` as tool output after user/client approval, including granted permissions, scope, and strict auto review flag (`request_permissions.rs:64-99`, `request_permissions.rs:107-116`).

What the runtime does:
- Claude runtime evaluates rules and tool-specific permission checks before mode grants. It then performs mode transforms, hook/classifier/headless fallback, and finally queues a user-facing confirmation only when ask survives those gates (`permissions.ts:473-535`, `permissions.ts:1060-1320`, `useCanUseTool.tsx:27-182`).
- Claude runtime updates both in-memory permission context and persistent settings for supported update destinations when a user or hook accepts permission updates (`PermissionContext.ts:139-147`, `PermissionContext.ts:291-336`, `PermissionUpdate.ts:55-216`, `PermissionUpdate.ts:222-389`).
- Claude runtime treats some policy as non-overridable by user approval: managed-only policy restricts rule sources, dangerous auto-mode rules are stripped while auto is active, and safety checks/content ask rules remain bypass-immune (`permissionsLoader.ts:27-44`, `permissionSetup.ts:267-553`, `permissions.ts:1238-1260`).
- Codex runtime computes an approval requirement per tool call, optionally evaluates hooks/guardian, asks clients/users when required, records/caches session approvals, then selects sandbox/no-sandbox first attempt according to the active filesystem/network policies and additional permissions (`sandboxing.rs:39-116`, `sandboxing.rs:158-363`, `orchestrator.rs:132-235`).
- Codex runtime normalizes additional permissions and grants them through permission-request responses; later shell/apply-patch calls can use those granted overlays without requiring model-supplied sandbox args when tests authorize the request (`request_permissions.rs:81-116`, `request_permissions_tool.rs:206-315`, `request_permissions_tool.rs:333-430`).

What gets persisted:
- Claude persists user/project/local permission updates to settings when the update destination supports persistence; session updates remain in the runtime permission context. Policy, flag, and command sources are effectively read-only for ordinary permission editing (`PermissionUpdate.ts:139-216`, `PermissionUpdate.ts:222-389`, `permissionsLoader.ts:147-156`).
- Claude persists auto-mode denial tracking in app state or local subagent denial state, and mode transitions can strip and later restore dangerous persisted rules (`permissions.ts:963-978`, `permissionSetup.ts:505-579`).
- Codex keeps `ApprovedForSession` approvals in an in-memory `ApprovalStore` keyed by tool-specific serialized approval keys. Review decisions can propose exec-policy or network-policy amendments, but those amendment application paths are source-specific and should be traced separately before claiming durable policy-file persistence (`sandboxing.rs:39-116`, `approvals.rs:32-52`, `approvals.rs:78-83`, `protocol.rs:3788-3803`).
- Codex persists permission/profile identity into session/configuration surfaces such as configured-session events and app-server thread responses, but this card does not trace config-write APIs for named permission profiles (`protocol.rs:446-450`, `models.rs:458-485`, `thread_lifecycle.rs:638-666`).

Shared invariant:
- Permissions are a layered runtime enforcement system, not only a prompt. Both references separate model-visible guidance or tool schema from runtime checks, approval decisions, persisted or session grants, user/TUI display, and debug/eval traces.
- A deny/forbidden rule must win before a later allow path. Approval must be source-attributed, and reusable approvals must have explicit scope and keys rather than a global "user said yes" flag.
- Sandbox policy and permission policy are coupled but not identical: permission approval can change sandbox selection or grant additional permissions, but filesystem/network enforcement remains its own runtime layer.

Claude-specific design:
- Claude's permission surface is mode/rule based: `PermissionMode`, `PermissionRule`, `PermissionUpdate`, and tool-specific `checkPermissions` compose into a decision pipeline.
- Claude has source-specific modes and flows: `acceptEdits`, `bypassPermissions`, `dontAsk`, `plan`, feature-gated `auto`, managed-only policy mode, dangerous auto-mode rule stripping, PermissionRequest hooks, bridge/channel approval relays, and classifier auto-approval/denial tracking.
- Claude persists permission changes through settings destinations (`localSettings`, `userSettings`, `projectSettings`) and uses session rules/directories for non-persisted grants.

Codex-specific design:
- Codex's permission baseline is a typed `PermissionProfile` plus `AskForApproval` policy, with `AdditionalPermissionProfile` overlays for per-command/requested grants.
- Codex has source-specific approval artifacts: `ExecPolicyAmendment`, `NetworkPolicyAmendment`, guardian review IDs/events, `request_permissions`, app-server v2 approval requests, and tool-specific approval keys for shell/unified-exec/apply-patch.
- Codex's `GranularApprovalConfig` can reject entire approval categories before user display, which is not the same shape as Claude's mode/rule pipeline.

Distilled harness rule:
- Build permissions as a source-typed decision pipeline with separate layers for model-visible instructions, runtime checks, approval requests, session/persistent grants, display, and telemetry/tests. Every allow, deny, ask, forbidden, or approval-cache hit must be attributable to a concrete rule, mode, profile, policy, hook, user/client decision, or automated reviewer.

Anti-invention rule:
- Do not invent approval modes, persistence destinations, policy amendment semantics, or "safe command" rules. If following Claude, cite and preserve its mode/rule/update/checkPermissions pipeline. If following Codex, cite and preserve its PermissionProfile, AskForApproval, ExecApprovalRequirement, ApprovalStore, request_permissions, and app-server approval event model. If only one source implements a mechanism, mark it source-specific.

Verification pattern:
- For a Claude-style harness, verify: deny rules beat allow rules; content-specific asks and safety checks survive bypass; `dontAsk` converts ask to deny; headless runs hooks before auto-deny; user/hook accepts persist only supported destinations; managed-only policy hides/blocks ordinary allow persistence; auto mode strips/restores dangerous rules; classifier denial limits abort headless or fall back to prompt; channel/bridge replies do not become model turns.
- For a Codex-style harness, verify: permission profile compiles into filesystem/network enforcement; prompt instructions match active profile and approval policy; granular false gates auto-reject instead of prompting; shell/unified-exec/apply-patch approval keys cache only the intended scope; denied reads prevent no-sandbox escalation; request_permissions grants unblock later exec/apply_patch only within granted scope; app-server/TUI approval request shapes expose the same available decisions as protocol; telemetry/test traces record decision source.
