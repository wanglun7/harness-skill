Runtime trace evidence:
- Claude skills are prompt commands loaded from managed/user/project/additional/plugin/bundled/MCP/legacy sources. A skill is discovered as `skill-name/SKILL.md`, parsed for description, `when_to_use`, arguments, model, effort, `allowed-tools`, hooks, fork context, agent, shell, and model-invocation flags; only frontmatter/description is used for initial listing because full content is loaded on invocation (`loadSkillsDir.ts:67-105`, `loadSkillsDir.ts:155-264`, `loadSkillsDir.ts:403-480`).
- Claude skill invocation is a model-visible `Skill` tool. The prompt says to invoke a relevant skill before responding, lists available skills in system-reminder messages, respects budget truncation, and blocks duplicate invocation when the current turn already contains a command tag. Validation rejects unknown, disabled-for-model, or non-prompt commands; permission checking supports deny/allow rules and asks for explicit approval when no rule matches (`SkillTool/prompt.ts:20-196`, `SkillTool.ts:291-430`, `SkillTool.ts:432-578`).
- Claude inline skills are processed through `processPromptSlashCommand`, return new user/attachment/system messages tagged to the parent tool use, and may set turn-scoped allowed tools, model, or effort. Skills with `context: fork` prepare a child prompt from the skill body, parse `allowedTools`, choose the declared agent or general-purpose agent, and run a forked `runAgent` loop (`SkillTool.ts:580-780`, `forkedAgent.ts:143-231`, `SkillTool.ts:118-289`).
- Claude dynamic and rehydration surfaces are explicit. File/path-triggered dynamic skill directories can be discovered during a turn; `skill_listing` system reminders list newly available skills; `skill_discovery` tells the model which skills are relevant and how to invoke them; `invoked_skills` rehydrates full invoked skill content after resume/compact; UI-only `dynamic_skill` attachments do not become model messages (`attachments.ts:2544-2751`, `messages.ts:3503-3519`, `messages.ts:3644-3659`, `messages.ts:3723-3738`).
- Claude plugins are source-specific bundles, not a generic runtime abstraction. The loader recognizes marketplace, session-only, and built-in sources, finds `.claude-plugin/plugin.json`, auto-detects `commands/`, `agents/`, `skills/`, `output-styles/`, `hooks/`, validates manifest paths, applies enterprise marketplace allow/block policy fail-closed, merges session/marketplace/builtin plugins, demotes dependency failures session-locally, and caches enabled plugin settings (`pluginLoader.ts:1-33`, `pluginLoader.ts:1320-1610`, `pluginLoader.ts:1880-2025`, `pluginLoader.ts:3137-3225`).
- Claude plugin commands/skills become prompt commands named with plugin prefixes; plugin skill bodies receive `Base directory for this skill`, `${CLAUDE_PLUGIN_ROOT}`, `${CLAUDE_PLUGIN_DATA}`, `${CLAUDE_SKILL_DIR}`, `${CLAUDE_SESSION_ID}`, argument substitution, non-sensitive user config substitution, and shell expansion under declared allowed tools (`loadPluginCommands.ts:45-97`, `loadPluginCommands.ts:218-402`, `loadPluginCommands.ts:414-520`).
- Claude plugin agents become `AgentDefinition`s with namespaced agent types, tools/skills/model/background/isolation/effort/maxTurns, optional memory prompt integration, and explicit refusal to parse per-agent `permissionMode`, hooks, or MCP servers because those escalation surfaces must live at plugin manifest or local `.claude/agents` trust boundaries (`loadPluginAgents.ts:65-229`, `loadPluginAgents.ts:231-348`).
- Claude plugin hooks are loaded from enabled plugins, converted into native hook matchers with plugin root/name/id, atomically swapped into registered hooks, pruned when plugins are removed, and hot-reloaded when plugin-affecting settings change (`loadPluginHooks.ts:25-157`, `loadPluginHooks.ts:255-287`).
- Claude custom instructions are CLAUDE.md/rules files loaded as "memory" but prompted as codebase/user instructions. Loading order is managed, user, project, local; project discovery walks original cwd upward, reads `CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/rules/*.md`, handles additional directories behind an env gate, and fires `InstructionsLoaded` hooks only for real context loads, not external-include approval checks (`claudemd.ts:1-26`, `claudemd.ts:89-96`, `claudemd.ts:790-940`, `claudemd.ts:1038-1130`).
- Claude `@include` processing recursively resolves includes, tracks processed paths/depth, skips excluded files, includes external paths only when approved, lists external includes by parent path, and shows a warning dialog that persists project config as approved or declined (`claudemd.ts:615-685`, `claudemd.ts:1399-1430`, `ClaudeMdExternalIncludesDialog.tsx:14-137`, `Settings/Config.tsx:976-990`, `Settings/Config.tsx:1521-1532`).
- Codex skills are metadata/catalog entries parsed from `SKILL.md` plus optional `agents/openai.yaml`. Metadata includes name, description, interface, dependencies, policy, path, scope, and `plugin_id`; policy can disable implicit invocation or restrict products. Loader scans admin/system/user/repo/plugin/extra roots, dedupes by canonical path, sorts by scope/name/path, limits scan depth/dir count, and fails open on optional metadata parse failures (`model.rs:14-58`, `loader.rs:155-233`, `loader.rs:283-397`, `loader.rs:488-705`, `loader.rs:730-920`).
- Codex skill loading is per config/turn. Plugins contribute effective skill roots before skill loading; `SkillsManager` caches by cwd or config skill cache key, installs/removes bundled system skills, applies product filtering and user/session `skills.config` enable/disable rules by name or path (`session/turn_context.rs:777-791`, `skills.rs:36-46`, `manager.rs:60-194`, `config_rules.rs:13-128`).
- Codex model-visible skill listing is rendered as developer context. The renderer uses a default 8k character budget or 2% of model context window, may choose aliased paths to fit more metadata, emits warnings when descriptions are shortened/removed, and includes exact "How to use skills" instructions requiring the agent to read the full `SKILL.md` and referenced assets/scripts before acting (`render.rs:17-60`, `render.rs:62-200`, `available_skills_instructions.rs:1-49`, `session/mod.rs:2952-2973`).
- Codex explicit skill use is turn-scoped. Structured `UserInput::Skill` selections resolve first by path; text `$skill` mentions are resolved later with ambiguity checks, disabled-path checks, connector-name conflict checks, and de-dupe by path/name. Mentioned skills are read through their owning filesystem, tracked as analytics/OTel explicit invocations, converted into `<skill>` contextual user fragments, and recorded into conversation items before sampling. Extension-injected host skill prompts are deduped against legacy injection (`injection.rs:19-111`, `injection.rs:133-200`, `skill_instructions.rs:1-40`, `turn.rs:465-625`).
- Codex skills extension is source-specific. It contributes available-skill prompt fragments, injects selected main prompts as contextual user fragments, and exposes `skills.list`/`skills.read` only when an orchestrator provider is configured and enabled. `skills.list` returns opaque package/main-resource handles for orchestrator-owned skills; `skills.read` validates bounded handles, ensures the package is currently available, routes by authority/provider, and returns external-context JSON (`ext/skills/extension.rs:53-140`, `ext/skills/extension.rs:170-379`, `tools/mod.rs:33-160`, `tools/list.rs:26-112`, `tools/read.rs:26-112`, `sources.rs:72-218`).
- Codex plugins are inert resolved packages with manifest paths/resources tied to an environment authority. Manifests declare skills, MCP servers, apps, hooks, interface metadata, default prompts, and assets; resource mapping rejects paths outside the package root. Runtime outcomes expose only active plugins and derive effective skill roots, MCP servers, apps, hook sources, warnings, and prompt-safe capability summaries (`plugin/manifest.rs:3-57`, `plugin/provider.rs:8-124`, `load_outcome.rs:15-188`).
- Codex model-visible plugin instructions are capability guidance, not a plugin execution API. The developer context says plugins are bundles of skills/MCP/apps, skills are prefixed by plugin name, MCP tools retain normal names, and explicit plugin mentions should bias toward associated capabilities. Explicit `plugin://` or linked `@` mentions create turn-scoped developer hints listing that plugin's available MCP servers, enabled apps, and skill prefix (`available_plugins_instructions.rs:1-58`, `plugins/mentions.rs:62-103`, `plugins/injection.rs:14-59`, `plugins/render.rs:12-58`).
- Codex recommended plugin installation is a separate source-specific path. Recommended plugins can be injected as user context; model tools `list_available_plugins_to_install` and `request_plugin_install` have strict descriptions limiting use to explicit missing-plugin requests or recommended-plugin suggestions, and request-plugin-install builds an MCP elicitation payload with approval metadata and persistent install intent (`recommended_plugins_instructions.rs:1-49`, `list_available_plugins_to_install_spec.rs:1-20`, `request_plugin_install_spec.rs:10-80`, `request_plugin_install.rs:16-128`).
- Codex AGENTS.md instructions are discovered from project root to cwd without walking past the root. Candidate files prefer `AGENTS.override.md`, then `AGENTS.md`, then configured fallbacks; contents are bounded by `project_doc_max_bytes`, combined with host-provided user instructions and optional internal hierarchical-agent guidance, and rendered as a contextual user fragment under `# AGENTS.md instructions` with `<INSTRUCTIONS>` tags (`agents_md.rs:1-17`, `agents_md.rs:37-88`, `agents_md.rs:96-140`, `agents_md.rs:168-272`, `agents_md.rs:274-450`, `user_instructions.rs:1-29`, `session/mod.rs:3030-3032`).

What the model sees:
- Claude sees a model-visible `Skill` tool, system-reminder skill listings/discovery hints, and full skill prompt content only after invocation. Inline skill expansion appears as new turn messages; forked skills run as child agents and return result text rather than injecting the whole child transcript.
- Claude sees plugin-provided commands/skills/agents only after plugin loaders convert plugin files into normal commands/agents/hooks. The model does not see a generic "plugin call" surface; it sees plugin-namespaced skills/commands and hook/tool effects.
- Claude sees CLAUDE.md/rules as instruction context introduced by the memory instruction prompt; external included files appear only after project approval.
- Codex sees developer-context skill and plugin instructions at context assembly, explicit `<skill>` user fragments for mentioned skills, optional `skills.list`/`skills.read` namespace tools for orchestrator skills, `<recommended_plugins>` user context when recommendation candidates exist, and `AGENTS.md` instructions as contextual user content.

What the runtime does:
- Claude loads and caches skill/command/plugin inventories, watches skill changes in the REPL, merges dynamic skills, validates Skill tool input/permission, then either transforms a skill into prompt messages or runs it in a forked agent with scoped allowed tools/model/effort.
- Claude plugin runtime resolves manifests, marketplace policy, session/builtin/plugin-source merging, dependency demotion, command/skill/agent/hook conversion, plugin settings cache, and plugin hook hot reload.
- Claude instruction runtime recursively resolves CLAUDE.md/rules includes, gates external project includes through project config, and fires `InstructionsLoaded` hooks only when instruction files are actually loaded into context.
- Codex loads plugin outcomes before skill roots, loads skills through `SkillsManager`, assembles developer/user context sections, parses explicit skill/plugin/app mentions before sampling, records injected skill/plugin instruction items into conversation history, and routes orchestrator skill tools through providers.

What gets persisted:
- Claude persists skills and instructions as filesystem files (`SKILL.md`, command markdown, plugin manifests/components, CLAUDE.md/rules). It persists external CLAUDE.md include approval/decline in project config, caches plugin settings for synchronous access, and records injected skill/plugin-derived messages in session transcript through normal message flow.
- Claude dynamic skill sent-state and invoked skill state are process/session surfaces used to avoid duplicate listings and to rehydrate invoked skill content after resume/compact.
- Codex persists plugin configuration/selected capability roots through its config/state layers, skill enable/disable rules under `skills.config`, plugin-derived skill roots as runtime outcomes, and injected skill/plugin/AGENTS fragments as recorded conversation items.
- Codex AGENTS.md itself remains filesystem project/user content; the model-visible assembled instructions are contextual user fragments in conversation history.

What the user/TUI sees:

Shared invariant:
- Skills/plugins/custom instructions are model-context mechanisms first and runtime mechanisms second. Both references keep a strict projection boundary: a short model-visible catalog or instruction fragment, a separate runtime loader/dispatcher, source-specific persisted files/config, user/TUI affordances, and debug/eval evidence.
- Neither reference treats plugin as a single callable abstraction. Plugins contribute concrete capabilities: skills, commands, agents, MCP servers, apps, hooks, or instructions. Reuse must preserve the contributed capability type and its trust boundary.

Claude-specific design:
- Claude skills are executable prompt commands behind a `Skill` tool; the tool may inline prompt messages into the main conversation or fork a child agent based on skill metadata.
- Claude plugins can contribute commands, skills, agents, hooks, output styles, and user-config substitutions through `.claude-plugin/plugin.json` or marketplace metadata. Plugin agent frontmatter is intentionally less powerful than local agent frontmatter for escalation-sensitive fields.
- Claude custom instructions use CLAUDE.md/rules loading, `@include`, external-include approval, and `InstructionsLoaded` hooks.

Codex-specific design:
- Codex skills are cataloged `SKILL.md` resources. The main reusable path is progressive disclosure: list metadata first, inject/read full skill content only when selected or mentioned.
- Codex plugin instructions are capability guidance plus plugin-derived skill roots/MCP/apps/hooks; explicit plugin mention generates a turn-scoped hint rather than calling a plugin.
- Codex custom instructions are AGENTS.md/AGENTS.override.md hierarchical project docs rendered as contextual user fragments, with optional host-provided user instructions and internal hierarchical-agent guidance.
- Codex orchestrator skills expose `skills.list` and `skills.read` namespace tools with opaque authority/package/resource handles; this is not present in Claude's cited implementation.

Gaps or unknowns:
- Claude plugin installation/removal UI commands and full marketplace refresh flows are not exhaustively traced here; this card cites the loader/cache/contribution path required for harness design.
- Codex plugin manager internals beyond manifest/outcome/effective capability projection are not fully expanded; the exact remote marketplace/cache loading code is outside the cited mechanism surface.
- Codex AGENTS.md include semantics are not analogous to Claude `@include`; no custom include approval mechanism was found in the cited AGENTS.md path.
- Claude MCP-loaded skills are referenced through command filtering and `loadedFrom: 'mcp'`, but the full MCP prompt source path is not deeply traced in this card.

Distilled harness rule:
- Implement skills/plugins/custom instructions as separate source-proven projections. For every capability, cite where it is discovered, where metadata is projected to the model, when full instructions enter the turn, how invocation/read permissions are checked, what persistent config/files back it, what the user sees, and how logs/tests prove the behavior. Do not generalize a plugin into a callable entity unless the source has such a call surface.

Anti-invention rule:
- Do not invent skill frontmatter, plugin manifests, include syntax, invocation triggers, or validation strategy. If copying Claude, keep Skill-tool prompt-command semantics, plugin command/agent/hook loaders, and CLAUDE.md external include approval source-specific. If copying Codex, keep metadata-first skill catalogs, explicit mention injection, orchestrator `skills.list/read`, plugin capability hints, and AGENTS.md contextual user fragments source-specific.

Verification pattern:
- For Claude-style skills/plugins/instructions, verify: `SKILL.md` discovery and metadata parsing; skill listing budget and system-reminder projection; Skill tool validation and permission denial/approval; inline skill message injection; forked skill execution with allowed tools and agent selection; plugin manifest path validation and policy blocking; plugin command/skill/agent/hook conversion; plugin hook reload; CLAUDE.md include recursion and external include approval state.
- For Codex-style skills/plugins/instructions, verify: plugin outcome contributes effective skill roots before skill loading; skill roots cover admin/system/user/repo/plugin/extra scopes; disabled skill rules apply by path/name; available skills render within budget and warn on truncation; explicit structured/text skill mentions inject full `<skill>` content once; orchestrator `skills.list/read` validates handles and package availability; plugin mention injects capability hints only for available MCP/apps/skill prefix; AGENTS.md search stops at project root and renders through `<INSTRUCTIONS>`.

Skill text candidate:

For each row, cite exact source files/functions and fill: discovery source, metadata fields, model-visible catalog text, full-content trigger, runtime dispatcher or provider, permission/trust gate, persisted state, user/TUI surface, and debug/eval proof. Mark Claude SkillTool, Claude plugins, Claude CLAUDE.md includes, Codex skill catalogs, Codex orchestrator skill tools, Codex plugin capability hints, and Codex AGENTS.md as separate source-specific mechanisms unless both reference sources implement the same behavior.
