Shared invariant:
- Delegation is not just "call another model." In both references, a delegation mechanism has a model-visible schema, a runtime child context/session, a communication/result projection back to the parent, persisted child identity/history, user-visible status, and debug/eval traces. Forking is a specialized delegation mode that carries parent context under explicit source rules.

Claude-specific design:
- Claude's primary subagent mechanism is a tool-local in-process query runner (`runAgent`) with sidechain transcript persistence. It can run sync or async, and async background agents are surfaced through local task state and notifications.
- Claude's fork-subagent path is implicit: omitting `subagent_type` under a feature gate selects a synthetic fork agent, inherits the parent's rendered system prompt and exact tools, and constructs cache-stable fork messages with placeholder tool results.
- Claude has a source-specific teammate/swarm layer: `TeamCreateTool` plus AgentTool `name`/`team_name` spawns either out-of-process CLI teammates or in-process teammates, with team files, mailbox startup for external panes, and task-state tracking.
- Claude also has a generic `runForkedAgent` helper outside AgentTool for cache-safe forked query loops; it is source-specific and should not be generalized into the Agent tool without citing the callers.

Codex-specific design:
- Codex's primary subagent mechanism is thread-based. `AgentControl` is shared across the root thread tree and manages live thread registry, task paths, spawn edges, residency, execution capacity, status, messaging, and completion forwarding.
- Codex V2 delegation is path-based: each child has a canonical `AgentPath` under `/root`, and tools address agents by relative or canonical path instead of opaque agent IDs.
- Codex V2 fork semantics are explicit via `fork_turns`, implemented by stored-rollout history fork/truncation/filtering. Full-history fork forbids agent type/model/reasoning overrides in the handler (`multi_agents_v2/spawn.rs:67-85`; `multi_agents_common.rs:237-247`).
- Codex has a source-specific app-server `thread/fork` API that forks user threads from stored history with config overrides. This is separate from model-visible subagent delegation and must be marked as app-server/client behavior.

Distilled harness rule:
- Model delegation as a first-class harness subsystem with six separately cited planes: model schema/prompt, child context/session construction, parent-child communication, persisted child identity/history, user-visible status/result projection, and trace/eval evidence. Treat fork as a delegation subtype only when the source shows how parent context is copied or referenced.

Anti-invention rule:
- Do not invent a generic "subagent framework" from a single source. If delegation is tool-local sidechain execution, call it that; if it is thread-tree AgentControl, call it that; if it is teammate/swarm or app-server thread fork, mark it source-specific. Never claim fork context sharing unless the exact copied/truncated messages or rollout-history path is cited.

Verification pattern:
- For each delegation mechanism, verify in order: tool exposure and schema, model prompt guidance, child identity/session source, child context/system/tools construction, context-fork rule if any, dispatch/lifecycle path, parent-child message/result path, persisted transcript/thread/edge state, user/TUI status event, and debug/eval evidence.
- For fork mechanisms, additionally verify the parent-context input, filtering/truncation/repair behavior, cache or identity invariants, recursive-fork guard, and whether model/tool overrides are allowed.

Skill text candidate:
- When comparing subagent, fork, or delegation harnesses, keep the planes separate. First cite what the model can invoke. Then trace the runtime child creation path. Then trace how results and messages return to the parent. Then trace what persists and what the user sees. Only after those citations should you distill a reusable rule, and source-specific mechanisms must stay source-specific.
