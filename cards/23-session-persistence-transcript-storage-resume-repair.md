Runtime trace evidence:
- Claude persists session transcript entries as JSONL `SerializedMessage`/`LogOption` records with `sessionId`, `cwd`, `timestamp`, `version`, `gitBranch`, parent links, sidechain flags, agent/team identity, content replacements, worktree state, file history, attribution, mode, PR, and context-collapse metadata (`logs.ts:8-53`, `logs.ts:143-231`).
- Claude transcript files live under a project directory derived from cwd/session id; subagents use per-agent transcript paths and `.meta.json` sidecars so agent type, worktree, and description do not require JSONL schema changes (`sessionStorage.ts:198-303`).
- Claude writes lazily. Persistence can be skipped by tests/env/settings, the session file materializes only on first user/assistant message, buffered metadata is flushed after materialization, and chain-participant messages update `parentUuid`, `logicalParentUuid`, sidechain/team/agent fields, and last-prompt cache (`sessionStorage.ts:953-1075`).
- Claude appends metadata/content-replacement entries separately from transcript messages, dedupes sidechain writes against the right file, persists main-session records remotely, records sidechain transcript records, resets/adopts file pointers on resume/fork, and can hydrate local JSONL from remote/CCR foreground and subagent event streams (`sessionStorage.ts:1128-1668`).
- Claude resume loading uses lite head/tail reads for session lists, then full loading for the selected log. Full loading scans pre-boundary metadata, prefilters dead fork branches before parse, repairs legacy progress bridges, applies preserved-segment relinks and snip removals, reconstructs leaves, recovers parallel tool results, and emits consistency/cycle diagnostics (`sessionStoragePortable.ts:92-217`, `sessionStorage.ts:1770-1995`, `sessionStorage.ts:2265-2484`, `sessionStorage.ts:3150-3812`).
- Claude `/resume` and `continue` are local JSX commands. They list resumable sessions, support current-repo/all-project filtering, exact UUID matching, title/search fallback, cross-project command copying, and direct fallback to `getLastSessionLog` when enriched logs filter out an exact id (`resume.tsx:80-260`).
- Claude resume restores more than messages: file history, attribution, context-collapse state, TodoWrite state, custom agent/model override, mode, worktree cwd if it still exists, cost state, content replacements, transcript file pointer, and skill/conversation state (`sessionRestore.ts:64-520`, `conversationRecovery.ts:375-520`).
- Claude conversation recovery filters unresolved tool uses, orphaned thinking, and whitespace-only content; detects interrupted turns from the terminal assistant/user/tool-result shape; migrates legacy attachments; restores skill state; and can resume by latest session, explicit JSONL path, session id, or `LogOption` (`conversationRecovery.ts:148-520`).
- Claude subagent resume loads the agent transcript and metadata, filters invalid fragments, rebuilds subagent content-replacement state, checks the saved worktree path, reconstructs fork/general-purpose agent parameters, and starts a new background agent with prior messages plus a new user prompt (`resumeAgent.ts:58-180`).
- Codex persists a typed rollout stream. `SessionMeta` captures thread/session identity, fork/parent ids, cwd, originator, cli version, model provider, base instructions, dynamic tools, memory mode, agent metadata, and multi-agent version; `RolloutItem` records session metadata, response items, inter-agent communication, compaction, turn context, and event messages (`protocol.rs:2863-3045`).
- Codex `SessionConfiguredEvent` is the user/client bootstrap surface. It includes session/thread ids, model/provider/config, permission profile, initial messages for resumed sessions, and rollout path, with compatibility deserialization for older sandbox fields and missing thread ids (`protocol.rs:3591-3725`).
- Codex `RolloutRecorder` stores inspectable JSONL, creates new rollout metadata lazily, opens resumed rollouts for append, writes through a bounded background writer, keeps pending items after failed persist/flush for retry, and loads rollout items while warning on parse errors and skipping legacy ghost snapshot lines (`recorder.rs:66-96`, `recorder.rs:687-980`).
- Codex repairs session lists through state-db/filesystem reconciliation. Listing and latest-thread lookup can start from SQLite, overfetch and repair stale or missing rows from rollout files, fall back to filesystem scans, and skip stale DB-only rows whose rollout no longer exists (`recorder.rs:360-590`, `metadata.rs:25-220`, `session_index.rs:17-260`).
- Codex resume reconstruction is not raw replay. `reconstruct_history_from_rollout` scans rollout items backward to find the newest compaction/replacement-history baseline and previous turn settings, then materializes forward into `ContextManager`, respecting rollback markers, compaction windows, turn-context reference items, and legacy compaction rebuild (`rollout_reconstruction.rs:4-320`).
- Codex `record_initial_history` treats new/cleared/resumed/forked history differently: resumed sessions reconstruct rollout history, warn on model mismatch, seed token usage, and flush; forked sessions persist initial history immediately and ensure the rollout is materialized (`session/mod.rs:1224-1310`).
- Codex records response items and inter-agent communication into both runtime state and rollout, and records `TurnContextItem` after real user turns so a later resume/lazy replay can recover the latest durable cwd/model/context baseline (`session/mod.rs:2658-2690`, `session/mod.rs:3104-3215`).
- Codex CLI `resume` resolves thread ids through app-server `thread/list` and `thread/resume`; TUI resume reads cwd/model first from state DB, falls back to rollout parsing, and prompts when the saved cwd differs from current cwd (`exec/src/lib.rs:782-840`, `tui/src/session_resume.rs:1-180`).
- Codex app-server resume has two paths: rejoin an already-running thread with listener attachment, or cold-resume from explicit history, stored thread metadata, or rollout. It rejects closing/archived/unmaterialized sessions, detects conflicting overrides, applies stored metadata, builds initial turns pages, emits thread resume response, and sends token usage/goal snapshots after resume (`thread_processor.rs:2500-3235`, `thread_processor.rs:3847-3867`).
- Codex remote app-server response redaction is display/API-only: MCP arguments/results/errors and generated images can be removed from the thread-resume payload while persisted rollout and model resume remain unchanged (`thread_resume_redaction.rs:1-44`).
- Codex also has a separate global user prompt history at `~/.codex/history.jsonl`; it is append-only, one JSON object per line, locked during append, and skipped when persistence is disabled. This is source-specific and separate from rollout transcript persistence (`message-history/src/lib.rs:1-15`, `message-history/src/lib.rs:83-150`).

What the model sees:
- Claude sees the reconstructed `Message[]` produced by transcript loading and conversation recovery, not raw JSONL lines. That reconstructed history has unresolved tool uses, orphaned thinking, invalid fragments, some whitespace-only content, and external-user REPL wrappers removed or normalized before it is used for resume (`sessionStorage.ts:4384-4461`, `conversationRecovery.ts:148-520`).
- Claude may see resume-specific synthetic continuation context when interruption detection determines the last recorded turn did not complete. The exact injected shape is governed by `deserializeMessagesWithInterruptDetection`, not by a generic resume rule (`conversationRecovery.ts:148-373`).
- Claude resumed subagents see their filtered agent transcript plus a new user prompt, with fork/general-purpose parameters reconstructed from agent metadata and saved transcript state (`resumeAgent.ts:58-180`).
- Codex sees reconstructed `ResponseItem` history plus context-manager state rebuilt from rollout items, compaction replacement history, and latest durable turn context. `ContextManager` normalizes API-history invariants before prompt construction, including call/output consistency and unsupported-image stripping (`rollout_reconstruction.rs:4-320`, `context_manager/history.rs:34-100`, `context_manager/history.rs:312-407`).
- Codex clients see `SessionConfiguredEvent.initial_messages` for resumed sessions, but model prompt history is still the reconstructed core history, not the UI bootstrap event itself (`protocol.rs:3591-3725`, `session/session.rs:1040-1195`).

What the runtime does:
- Claude runtime separates cheap session listing from full resume: lite reads extract first prompt/head-tail metadata; full resume loads the selected chain, applies repair transforms, restores runtime side state, and adopts or resets the transcript file pointer before continuing (`sessionStoragePortable.ts:92-217`, `sessionStorage.ts:3869-3932`, `sessionRestore.ts:405-520`).
- Claude runtime uses parent/leaf chain semantics. It can choose the latest non-sidechain session leaf, rebuild a conversation chain, detect consistency deltas, repair parallel tool results, and keep subagent sidechain transcript state in separate files (`sessionStorage.ts:2265-2484`, `sessionStorage.ts:3869-4236`).
- Claude runtime treats resume as state restoration, not only message replay: worktree cwd, file history, attribution, TodoWrite state, mode, cost, content replacements, context-collapse state, custom agent state, and skill state are restored from transcript metadata when available (`sessionRestore.ts:64-520`, `conversationRecovery.ts:375-520`).
- Codex runtime treats rollout as the canonical durable stream and state DB/index as discoverability/cache layers that can be repaired from rollout files (`recorder.rs:360-590`, `metadata.rs:25-220`, `session_index.rs:17-260`).
- Codex runtime chooses between running-thread rejoin and cold resume. Running-thread resume attaches a listener and returns existing history/config snapshots; cold resume constructs `InitialHistory::Resumed` from rollout, initializes a session, dispatches `SessionConfiguredEvent`, and records initial history into the new runtime state (`thread_processor.rs:2500-3235`, `session/session.rs:500-660`, `session/session.rs:1040-1195`, `session/mod.rs:1224-1310`).

What gets persisted:

- Codex persists rollout JSONL items, compressed/materialized rollout append state, state-db thread metadata, session index entries, archived-session state, per-turn context baselines, response items, inter-agent communication records, compaction records, event messages, and separate global prompt history (`protocol.rs:2863-3045`, `recorder.rs:66-96`, `metadata.rs:25-220`, `session_index.rs:17-260`, `message-history/src/lib.rs:1-150`).
- Codex app-server response redaction does not alter persisted rollout or model-resume content; it only changes thread-resume payloads sent to remote clients (`thread_resume_redaction.rs:1-44`).

Shared invariant:
- A top-tier coding-agent harness treats the durable transcript as an append-oriented, source-typed event/message log, and treats resume as source-defined reconstruction plus repair, not raw file replay.
- Session identity, cwd/worktree context, runtime side state, model-visible history, user/TUI list metadata, and debug/eval evidence are separate surfaces. Both references optimize session listing with lightweight metadata paths and reserve full repair/reconstruction for actual resume.
- Context baselines and compaction state must be persisted in a form that the loader can use after process death; do not rely on in-memory context-manager state alone.

Claude-specific design:
- Claude uses JSONL transcript records with `uuid`/`parentUuid`/`logicalParentUuid`, latest-leaf chain reconstruction, sidechain flags, per-agent transcript files, and `.meta.json` subagent sidecars.
- Claude re-appends or reads metadata records such as file history, attribution, worktree state, mode, context-collapse data, content replacements, and TodoWrite state to reconstruct runtime behavior beyond the message list.
- Claude resume includes legacy transcript repair paths: progress-message bridging, dead fork branch filtering before parse, preserved-segment relinks, snip removals, parallel tool-result recovery, and external-user transcript wrapper stripping.
- Claude remote/CCR hydrate is source-specific; it is not a general requirement for a harness unless the target has equivalent remote transcript persistence.

Codex-specific design:
- Codex uses typed `RolloutItem` streams, `SessionMeta`, `TurnContextItem`, `CompactedItem`, and `SessionConfiguredEvent` rather than Claude-style parent-linked transcript messages.
- Codex reconstruction scans rollout items backward to find the newest durable baseline, then materializes forward into `ContextManager` with replacement history, rollback handling, turn-context baselines, and compaction windows.
- Codex uses state DB and session index as repairable discovery layers over rollout files; rollout remains the durable source for cold resume.
- Codex app-server can rejoin running threads or cold-resume stored rollouts, and can redact response payloads for remote clients without changing persisted or model-resume state.
- Codex global `history.jsonl` is prompt-history persistence, not conversation transcript persistence.

Distilled harness rule:
- Build session persistence around a typed append-only durable stream, a cheap listing/index layer, and a source-defined reconstruction loader. The loader must restore model-visible history, runtime side state, persisted context baselines, display metadata, and debug/eval repair evidence as separate outputs. If a mechanism exists only in Claude or only in Codex, keep it source-specific.

Anti-invention rule:
- Do not invent a transcript schema, resume database, repair pass, or compaction baseline. If following Claude, keep JSONL parent-chain semantics, sidechain files, metadata records, and Claude-specific repair transforms. If following Codex, keep RolloutItems, ContextManager reconstruction, TurnContext baselines, state-db/filesystem repair, and app-server running-thread versus cold-resume split. Do not claim a shared subagent transcript model, redaction model, or remote hydrate model unless both sources implement the same behavior.

Verification pattern:
- For a Claude-style harness, verify: session file materializes only after recordable user/assistant messages; lite session listing extracts title/first prompt without full parse; resume by exact id/title/search loads the selected chain; missing worktree is not restored; file history/attribution/todo/mode/content replacements/context collapse restore; unresolved tools and interrupted turns are repaired; sidechain subagent resume uses the agent transcript/sidecar; content replacements replay; remote hydrate can rebuild foreground and subagent JSONL.
- For a Codex-style harness, verify: rollout recorder persists idempotently and retries failed writes; parse errors and legacy ghost snapshots do not abort load; state-db/index repair handles stale and missing rollout paths; resume by id appends to the same rollout file; `resume --last` respects cwd via latest turn context; CLI overrides are preserved or warned exactly as source says; app-server rejects unmaterialized/archived sessions; running-thread resume reattaches instead of cold-starting; initial turns page, redaction, token usage, and goal snapshots are emitted after resume.
