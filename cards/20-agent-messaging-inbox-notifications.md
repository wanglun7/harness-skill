Runtime trace evidence:
- Claude teammate messages are file-backed. `writeToMailbox` resolves `.claude/teams/<team>/inboxes/<agent>.json`, creates the file if needed, lock-re-reads, appends a `read:false` message, writes JSON, and releases the lock; `readUnreadMessages` filters unread messages and `markMessagesAsRead` marks them under a file lock (`teammateMailbox.ts:52-66`, `teammateMailbox.ts:79-125`, `teammateMailbox.ts:127-192`, `teammateMailbox.ts:273-342`).
- Claude separates plain teammate content from structured protocol messages. The classifiers identify idle, permission, sandbox, plan approval, shutdown, task assignment, team permission, and mode-set messages; `isStructuredProtocolMessage` keeps protocol messages routed by `useInboxPoller` rather than consumed as raw LLM context (`teammateMailbox.ts:391-1095`).
- Claude `useInboxPoller` polls every second for the current agent/team-lead inbox, routes permission/sandbox/plan/shutdown/mode messages to their handlers, queues busy-session regular messages into `AppState.inbox`, submits idle regular messages immediately, and marks file messages read only after submit or reliable queueing (`useInboxPoller.ts:74-125`, `useInboxPoller.ts:139-248`, `useInboxPoller.ts:250-864`).
- Claude regular teammate messages reach the model as `<teammate-message teammate_id="...">...</teammate-message>` either by immediate `onSubmitTeammateMessage`, delayed idle delivery from `AppState.inbox`, or attachment generation. Attachment generation filters structured protocol messages, deduplicates file/AppState copies, collapses duplicate idle notifications, builds a `teammate_mailbox` attachment, then marks only non-structured file messages read (`useInboxPoller.ts:810-864`, `useInboxPoller.ts:917-928`, `attachments.ts:3532-3715`, `messages.ts:3453-3466`).
- Claude `SendMessage` is the model-visible send path for teammate inboxes. Plain string messages call `writeToMailbox` with sender name, color, summary, timestamp, and content; structured messages dispatch to shutdown or plan-approval helpers; stopped background agents may be resumed with the sent message (`SendMessageTool.ts:149-189`, `SendMessageTool.ts:520-640`, `SendMessageTool.ts:820-917`).
- Claude in-process teammates do not rely only on file polling. Transcript-viewer input appends `pendingUserMessages`, `waitForNextPromptOrShutdown` polls that in-memory queue, and non-user messages are wrapped in the same `<teammate-message>` XML before becoming the next prompt (`InProcessTeammateTask.tsx:47-84`, `inProcessRunner.ts:689-735`, `inProcessRunner.ts:1353-1408`).
- Claude background local-agent completion uses a separate command queue. `enqueueAgentNotification` constructs `<task-notification>` XML with status, summary, result, output path, usage, and worktree fields, then enqueues a non-editable `task-notification`; queue operations are logged with `recordQueueOperation` (`LocalAgentTask.tsx:197-261`, `messageQueueManager.ts:28-60`, `messageQueueManager.ts:137-149`, `messageQueueManager.ts:343-375`, `sessionStorage.ts:1208-1221`, `sessionStorage.ts:1464-1466`).
- Codex agent messages are typed protocol values, not file inbox rows. `InterAgentCommunication` carries `author`, `recipient`, optional other recipients, `content` or `encrypted_content`, optional metadata, and `trigger_turn`; `to_model_input_item` projects it as a `ResponseItem::AgentMessage`, using encrypted content with a typed prefix for `MESSAGE` versus `NEW_TASK` (`protocol.rs:677-770`).
- Codex `send_message` and `followup_task` share a handler. It resolves the target agent path, ensures the target is known and loaded, builds an `InterAgentCommunication`, applies `QueueOnly` for `send_message` or `TriggerTurn` for `followup_task`, sends it through `AgentControl`, emits `SubAgentActivityKind::Interacted`, and returns an empty successful tool output (`message_tool.rs:1-31`, `message_tool.rs:72-125`).
- Codex session delivery queues inter-agent messages in `InputQueue.mailbox_pending_mails`. `trigger_turn` calls `maybe_start_turn_for_pending_work_with_sub_id`; otherwise queued mail waits until accepted by the active turn or the next turn. `subscribe_activity` lets `wait_agent` observe mailbox or steered-user-input activity without returning message contents (`input_queue.rs:34-81`, `input_queue.rs:197-253`, `handlers.rs:277-292`, `wait.rs:69-110`, `wait.rs:171-189`).
- Codex child completion notification is synthesized as another inter-agent message. Terminal MultiAgentV2 child turns call `forward_child_completion_to_parent`, format a completion envelope, send it to the parent with `trigger_turn:false`, and optionally write rollout trace payload; `AgentControl` has a detached watcher path that does the same for thread-spawned children (`session/mod.rs:1699-1807`, `agent/control.rs:397-481`).

What the model sees:
- Claude sees `SendMessage` as an enabled swarm tool with `to` and message fields; plain string sends are model-visible as successful tool results such as "Message sent to X's inbox", while delivered teammate content is injected as user/meta text with `<teammate-message>` wrappers (`SendMessageTool.ts:520-640`, `SendMessageTool.ts:876-917`, `messages.ts:3453-3466`).
- Claude sees queued background task notifications as user-message text wrapped by origin: `wrapCommandText` turns `task-notification` origin into "A background agent completed a task:\n..." before the queued command becomes provider input (`messages.ts:3739-3765`, `messages.ts:5496-5511`).
- Codex sees `send_message`, `followup_task`, and `wait_agent` schemas. `send_message` queues a prompt delivery, `followup_task` may trigger a target turn, and `wait_agent` reports only wait outcome summaries, not mailbox contents (`multi_agents_spec.rs:156-215`, `multi_agents_spec.rs:255-265`, `wait.rs:126-160`).
- Codex receives mailbox content as `agent_message` input items. For encrypted messages the text prefix includes message type, task name, sender, and payload marker followed by encrypted content; for plaintext the content is the raw message (`protocol.rs:730-770`).

What the runtime does:
- Claude keeps mailbox delivery loss-aware: file messages are marked read only after immediate delivery or queueing, and attachment generation marks only non-structured messages after the attachment exists (`useInboxPoller.ts:843-864`, `attachments.ts:3676-3695`).
- Claude uses separate queues for different surfaces: file-backed teammate inboxes for cross-agent/team protocol, `AppState.inbox` for busy-session pending teammate messages, in-process task state for transcript-viewer input, and `commandQueue` for user input and task notifications (`teammateMailbox.ts:127-192`, `useInboxPoller.ts:822-858`, `InProcessTeammateTask.tsx:63-84`, `messageQueueManager.ts:41-60`).
- Codex keeps one session-scoped mailbox queue for inter-agent communication and a watch channel for waiters. `TurnInput::InterAgentCommunication` is appended to current pending input only when the active turn accepts mailbox delivery; otherwise it remains queued (`input_queue.rs:12-38`, `input_queue.rs:49-81`, `input_queue.rs:126-176`, `input_queue.rs:197-253`).
- Codex `followup_task` differs from `send_message` only by `trigger_turn:true`; the shared handler and queue path are otherwise the same (`message_tool.rs:1-31`, `message_tool.rs:58-125`).

What gets persisted:
- Claude file mailbox JSON persists teammate messages until they are marked read. Session JSONL persists command queue operations as `queue-operation` entries and normal transcript/sidechain messages through message-chain insertion (`teammateMailbox.ts:127-192`, `teammateMailbox.ts:273-342`, `sessionStorage.ts:1208-1228`, `sessionStorage.ts:1410-1466`).
- Claude `AppState.inbox`, in-process teammate pending messages, and command queue are runtime state; only their submitted/projection artifacts are later transcript-visible or queue-operation persisted by the cited paths.
- Codex persists inter-agent communication as `RolloutItem::InterAgentCommunication`, records its model-input equivalent in conversation history, and reconstructs it into history on rollout replay (`session/mod.rs:2673-2690`, `rollout_reconstruction.rs:223-228`, `rollout_reconstruction.rs:277-283`).
- Codex app-server thread history ignores raw rollout `InterAgentCommunication` items for UI rebuilding but converts `AgentMessage` events and collab/subagent activity events into thread items (`thread_history.rs:380-391`, `thread_history.rs:476-491`, `thread_history.rs:901-1002`).

Shared invariant:
- Agent messaging must preserve author/recipient identity, route control/protocol messages before exposing content as ordinary model context, and make delivery observable without requiring the sender to poll raw transcript state.
- Completion notifications are messages, not special hidden side effects: Claude uses `<task-notification>`/teammate structured JSON and Codex uses `InterAgentCommunication` completion envelopes.

Claude-specific design:
- Claude uses a file-backed team inbox plus JSON classifiers and XML model wrappers. This mailbox design, `AppState.inbox`, and `<teammate-message>`/`<task-notification>` tags are Claude-specific and should not be generalized to Codex.
- Claude has a separate in-memory `commandQueue` for user inputs, task notifications, and meta queued commands, with editable/visible policy and queue-operation persistence.

Codex-specific design:
- Codex uses typed protocol `InterAgentCommunication` and `ResponseItem::AgentMessage` rather than XML teammate wrappers or file inbox JSON.
- Codex `wait_agent` in MultiAgentV2 is an activity gate: it reports mailbox/steer/timeout status and intentionally does not return message contents; delivery happens through the next model request's agent-message inputs.

Gaps or unknowns:
- Claude evidence here does not prove a stable external schema for the file mailbox beyond the local `TeammateMessage` type and JSON file behavior.
- Codex app-server notification method names for all collab item updates were not separately traced in this card; the source-grounded surfaces used here are core `EventMsg`, app-server `ThreadItem`, and TUI rendering.
- Cross-machine Claude bridge/UDS inbox behavior is visible in `SendMessageTool` permission/validation branches but is not deeply traced here because mechanism 20 is limited to agent messaging/inbox/notifications.

Distilled harness rule:
- Treat agent messages as first-class inputs with explicit sender, recipient, delivery mode, and state transition. Separate protocol/control messages from ordinary prompt content, and define a single projection path for each surface: model input, runtime queue, persisted replay, TUI summary, and debug/eval trace.

Anti-invention rule:
- Do not invent an inbox abstraction that neither reference uses. If deriving from Claude, keep file inbox + XML wrappers + command queue source-specific. If deriving from Codex, keep typed `InterAgentCommunication` + `agent_message` input + `wait_agent` activity gate source-specific.

Verification pattern:
- For a Claude-style harness, verify: send writes an unread mailbox row; poller routes structured messages without raw model exposure; regular messages become exactly one `<teammate-message>` delivery; busy delivery survives until idle; background completion produces `<task-notification>` and a queue operation.
- For a Codex-style harness, verify: `send_message` enqueues mail without starting a turn; `followup_task` enqueues mail and triggers pending work; `wait_agent` wakes on mailbox or steered input without returning contents; child completion persists as inter-agent communication and replays as model history.

Skill text candidate:
- When studying or reusing agent messaging, first identify the transport: file mailbox, in-memory queue, protocol item, or transcript injection. Then trace five separate projections before naming the design: (1) what the sender tool writes, (2) how the receiver is awakened or later drained, (3) what exact item the model sees, (4) what persists for resume/replay, and (5) what the user/TUI sees. Mark XML/file-inbox designs as Claude-specific and typed `InterAgentCommunication`/`agent_message` designs as Codex-specific unless both sources show the same mechanism.
