Shared invariant:
- Memory is not a single cache. Both references separate at least four surfaces: model-visible memory guidance/context, runtime extraction/recall orchestration, persisted memory artifacts/state, and user/debug/eval visibility.
- Durable memory writes must be constrained by explicit storage roots and source-specific eligibility gates. Retrieval must either inject bounded content with stable provenance or provide tools/instructions for targeted lookup.

Claude-specific design:
- Claude's automatic memory is end-of-turn/fork-agent driven, file topic/index based, and can recall selected files into later turns as system-reminder attachments.
- Claude's session memory is a separate per-session markdown summary used to support compaction; it is not the same as cross-session auto-memory.
- Claude custom agents can own persistent scoped memory directories (`user`, `project`, `local`) and optional project snapshot sync.

Codex-specific design:
- Codex generates memories at startup from prior eligible rollouts, not as a normal end-of-turn hook in the active conversation.
- Codex uses a two-phase pipeline: phase1 extracts raw memories and rollout summaries into a memories DB; phase2 consolidates those into file-backed memory artifacts through an internal locked-down agent in `$CODEX_HOME/memories`.
- Codex exposes read-path memory as developer instructions plus optional namespaced tools, tracks citations back to rollout ids, and can mark a thread's memory mode polluted when external context is used.

Gaps or unknowns:
- Claude team memory internals are only cited where `loadMemoryPrompt` branches and extraction counts team paths; the team-sync implementation is not fully traced in this mechanism card.
- Claude tests for memory recall/extraction/session-memory were not separately enumerated; the card uses exported helpers and runtime logs as debug/eval surfaces.
- Codex final consolidation artifact schema is prompt/file-driven rather than typed in the cited code; `MEMORY.md`, `memory_summary.md`, `raw_memories.md`, and rollout summaries are known surfaces, but exact consolidation editing decisions are delegated to the phase2 agent prompt.
- Neither source shows a generic "memory abstraction" that covers every mode; any reusable skill must preserve source-specific distinctions.

Distilled harness rule:
- First classify a memory mechanism by lifecycle and surface: instruction memory, long-term project/user memory, session-summary memory, agent-scoped memory, read-path recall, write-path extraction, consolidation, citation/usage tracking, or reset/settings control. Then trace five projections before reusing it: what the model sees, what the runtime schedules, what path/database stores it, what the user/TUI sees, and what debug/eval evidence proves it.

Anti-invention rule:
- Do not rename all of these mechanisms "memory" and merge them. A Claude-derived harness may copy file-based auto-memory recall/extraction or session-memory compaction only with those exact gates and projections marked source-specific. A Codex-derived harness may copy startup phase1/phase2 rollout memory only with thread `memory_mode`, memories DB, file artifacts, and citation tracking marked source-specific.

Verification pattern:
- For Claude-style memory, verify: auto-memory path resolves from the documented precedence; `loadMemoryPrompt` produces model-visible save/access rules; relevant recall scans/selects/reads at most five bounded files and injects `relevant_memories`; extraction skips when the main agent already wrote memory; forked extraction can only edit/write memory roots; `memory_saved` renders; session-memory compaction falls back when summary is absent, empty, stale, or boundary is unknown.
- For Codex-style memory, verify: feature/config gates control `generate_memories` and `use_memories`; new and resumed threads persist the expected `memory_mode`; startup skips ephemeral/subagent sessions; phase1 claims only enabled, stale, eligible threads and records raw memory; phase2 materializes inputs, detects git-baseline diff, runs a locked-down consolidation agent, and updates job state; citations update stage1 usage; external context can mark memory polluted; reset clears memory artifacts and DB rows while preserving thread metadata.

Skill text candidate:
- When analyzing a coding-agent harness memory system, never start with a generic memory diagram. Instead, make a source-grounded table with rows for each memory lifecycle. Treat Claude's CLAUDE.md loading, auto-memory recall/extraction, session-memory compaction, and agent-memory scopes as separate mechanisms.
