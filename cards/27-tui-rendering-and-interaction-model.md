Runtime trace evidence:
- Claude REPL is the TUI orchestrator. Streaming assistant text is kept in React state, hidden when reduced-motion or terminal bugs apply, and only complete lines are shown while streaming (`REPL.tsx:1458-1473`). Cancel preserves partially streamed text by appending an assistant message before resetting state and aborting the active prompt, permission, remote, or local request path (`REPL.tsx:2106-2163`).
- Claude transcript mode is a distinct render branch. It freezes message/tool-use lengths on entry, optionally uses virtual scroll inside `AlternateScreen`/`FullscreenLayout`, mounts scroll keybindings before cancel handling so copy-selection can win over cancellation, and otherwise falls back to native terminal scrollback plus a capped transcript (`REPL.tsx:4380-4489`).
- Claude normal mode chooses displayed messages from viewed-agent state, synchronous messages, or deferred messages; renders a `PermissionRequest` overlay while a tool permission is focused; then composes `Messages`, auth/status rows, local JSX, dialogs, surveys, `PromptInput`, background hints, message actions, and selector surfaces (`REPL.tsx:4492-4519`, `REPL.tsx:4533-4565`, `REPL.tsx:4569-4907`).
- Claude `Messages` is not a passive list. It normalizes messages, synthesizes streaming tool-use messages with deterministic UUIDs to avoid Ink remount corruption, filters compact boundaries only where display needs it, keeps snipped messages for UI scrollback, applies grouping/collapse transforms, builds lookup maps, then either renders a virtual list or a capped flat list (`Messages.tsx:341-379`, `Messages.tsx:443-529`, `Messages.tsx:531-543`, `Messages.tsx:699-720`).
- Claude message rendering separates row dispatch from tool-owned display. `MessageImpl` switches assistant/user/system/renderable types, while `AssistantToolUseMessage` finds the tool, validates input, checks permission/progress state, and calls tool-provided renderers for queued/progress/use displays with render-error guards (`Message.tsx:98-245`, `AssistantToolUseMessage.tsx:51-293`, `AssistantToolUseMessage.tsx:304-367`).
- Claude prompt input owns submit gating and local interaction behavior: footer selection and agent-selection can block Enter; prompt suggestions/speculation can short-circuit normal query; direct member messages can route to mailbox; suggestions can block submission; active viewed agents receive input instead of the leader; text input receives history, image paste, cursor, ghost text, highlights, Vim/TextInput choice, footer, and clipped notifications (`PromptInput.tsx:984-1125`, `PromptInput.tsx:2172-2296`).
- Codex TUI runtime is a Ratatui event/render loop. `TuiEvent` separates key, paste, resize, and draw; `Tui` owns frame scheduling, terminal state, pending history lines, alternate-screen state, focus/notification state, terminal restore for external programs, desktop notifications, and synchronized drawing (`tui.rs:510-549`, `tui.rs:568-730`, `tui.rs:732-802`, `tui.rs:851-951`).
- Codex `App::run` builds `ChatWidget`, keymap, transcript state, app-server session state, and an event loop that selects over app events, active thread events, terminal events, and app-server events. Terminal draw/resize events call pre-render work, update transient UI state, render the chat widget with resize reflow, draw optional image overlays, and set cursor position from `ChatWidget` (`app.rs:1006-1056`, `app.rs:1102-1217`, `app.rs:1254-1350`).
- Codex global input lives above `ChatWidget` for app-level commands. It routes external editor launch through terminal restoration, toggles raw output and reflows transcript, enters alternate-screen transcript overlay, handles clear-screen reset, and passes ordinary keys to `ChatWidget` after global shortcuts/backtrack logic (`app/input.rs:1-89`, `app/input.rs:92-254`).
- Codex `ChatWidget` consumes protocol events and owns UI state, not agent execution. Source comments define committed transcript cells plus an in-flight active cell, transcript overlay live-tail caching, bottom-pane task-running derivation, and slash-command split; the struct contains stream controllers, running commands, hook cells, status state, input queue, keymap, pending notifications, terminal title/status caches, rollout/cwd/instruction display state, and composer state (`chatwidget.rs:1-32`, `chatwidget.rs:513-524`, `chatwidget.rs:525-734`).
- Codex protocol notifications update UI state directly: turn start/completion toggles task lifecycle; item start/completion routes command, patch, MCP, web-search, image, collab-agent, and subagent items; deltas feed agent message, plan, reasoning, terminal interaction, command output, and patch output handlers; replay flags suppress or alter some live effects (`chatwidget/protocol.rs:4-90`, `chatwidget/protocol.rs:92-226`, `chatwidget/protocol.rs:229-335`).
- Codex rendering composes a renderable tree rather than React components. `ChatWidget::as_renderable` stacks active transcript cell, active hook cell, pending usage/rate-limit cells, and bottom pane; `TranscriptAreaRenderable` asks each `HistoryCell` for width-aware display lines and scrolls overflow inside the allocated area; `ChatWidget` delegates render, desired height, cursor position, and cursor style to that renderable (`chatwidget/rendering.rs:5-60`, `chatwidget/rendering.rs:90-149`).
- Codex committed display is `HistoryCell`-based. Cells expose main display lines, raw copy-friendly lines, rich hyperlink lines, raw/rich mode selection, desired wrapped height, and transcript-overlay lines that may differ from main display; `App::insert_history_cell` pushes cells to retained transcript state, mirrors them into transcript overlay, inserts wrapped history lines into terminal scrollback, and schedules frames (`history_cell/mod.rs:180-280`, `app/history_ui.rs:11-36`).
- Codex bottom pane owns local focused interactions. It contains the `ChatComposer`, a stack of replacement views/popups, delayed approvals, pending input previews, thread approvals, context-window state, and a `StatusIndicatorWidget`; the status widget owns spinner timing, elapsed/interrupt text, optional details, frame scheduling for animation, and snapshot-tested rendering (`bottom_pane/mod.rs:1-15`, `bottom_pane/mod.rs:201-241`, `status_indicator_widget.rs:1-61`, `status_indicator_widget.rs:235-438`).

What the model sees:
- Claude model input is not the TUI tree. The model can receive user submissions and tool/permission outcomes from the runtime, but `Messages` filtering/collapse/virtualization, `StreamingMarkdown`, `PromptInputFooter`, terminal title, surveys, and dialogs are display surfaces unless their handlers explicitly submit text, route direct messages, resolve prompts, or update permission state (`Messages.tsx:475-529`, `Messages.tsx:677-720`, `PromptInput.tsx:984-1125`, `REPL.tsx:4519`).
- Claude UI can preserve partially streamed assistant text as a real assistant message on cancel; that changes local messages after user interruption, so a harness must distinguish display-only streaming preview from the message appended in the cancel path (`REPL.tsx:1458-1473`, `REPL.tsx:2121-2129`).
- Codex model input is not Ratatui state. `ChatWidget::submit_user_message_with_history_and_shell_escape_policy` converts text/images/mentions/skills/plugins/apps into `UserInput` items for app-server/core submission; `!cmd` is intercepted as a local shell command instead of model input (`chatwidget/input_submission.rs:98-183`, `chatwidget/input_submission.rs:185-280`).
- Codex `HistoryCell` display/raw/transcript lines and `StatusIndicatorWidget` rows are user-facing render artifacts, not provider payloads (`history_cell/mod.rs:180-280`, `status_indicator_widget.rs:235-300`).

What the runtime does:
- Claude runtime maintains UI state in React/Ink: streaming text/thinking/tool-use arrays, focused dialogs, tool permission queues, prompt queues, local JSX command surfaces, input mode, viewed-agent state, scroll refs, search/jump refs, and memoized message transforms (`REPL.tsx:1458-1473`, `REPL.tsx:4380-4489`, `REPL.tsx:4492-4519`, `Messages.tsx:443-529`).
- Claude runtime schedules interactive behavior through keybinding components and input handlers: transcript scroll handlers precede cancel handlers, chat submit is registered to avoid blocking autocomplete, and PromptInput toggles editor/stash/model/thinking/mode/image/global search/history behavior according to modal state (`REPL.tsx:4406-4428`, `PromptInput.tsx:1637-1737`).
- Codex runtime multiplexes app events, active thread events, terminal events, and app-server events in one loop, then schedules frames through `FrameRequester`; draw paths flush history lines above the viewport and re-render only the live viewport/bottom pane (`app.rs:1157-1217`, `tui.rs:785-802`, `tui.rs:851-951`, `app.rs:1254-1350`).
- Codex runtime keeps committed scrollback/history separate from active mutable display: terminal history insertion goes through `insert_history_cell`, active cells render in the main frame, and transcript overlay uses retained cells plus a live-tail cache key (`app/history_ui.rs:11-36`, `chatwidget.rs:1-17`, `chatwidget.rs:742-760`).

What gets persisted:
- Claude TUI state is mostly in-memory display state. Persisted side effects occur only when UI handlers call persistence paths, such as permission updates in sandbox/permission dialogs or global config updates from dialogs; message/session transcript persistence is a separate mechanism and not implied by rendering (`REPL.tsx:4677-4710`, `REPL.tsx:4750-4758`, `REPL.tsx:4911-4938`).
- Claude display can intentionally keep snipped/pre-compact messages for scrollback while model context uses compact-boundary filtering elsewhere; do not infer model persistence from UI history retention (`Messages.tsx:487-497`).
- Codex `App` retains `transcript_cells` and writes committed cells into terminal scrollback during the live TUI session; `/clear` resets overlay, transcript cells, deferred lines, reflow state, pending usage hints, backtrack, and skill warnings (`app.rs:521-528`, `app/history_ui.rs:11-36`, `app/history_ui.rs:128-178`).
- Codex durable session/rollout persistence is not the TUI render tree. The TUI tracks current rollout path, cwd, instruction-source paths, terminal title/status caches, and active thread ids for display/session coordination, but persistence semantics belong to the session/rollout mechanisms (`chatwidget.rs:682-689`, `app.rs:568-579`).

Shared invariant:
- TUI rendering is a separate harness layer from model context, runtime execution, transcript persistence, and debug/eval capture. It projects runtime events and state into interactive surfaces, but only specific input/approval/interrupt handlers feed decisions back into runtime/model channels.
- Live streaming display should have an explicit separation between provisional display and committed history/message state.
- Long transcripts require a bounded render strategy. Claude uses Ink virtual scroll or caps/fallback scrollback; Codex uses terminal scrollback plus retained `HistoryCell`s and active live cells.
- Interaction routing must be ordered and state-aware: modal/popup/input/search/scroll/cancel paths can conflict, so the source implementations encode precedence instead of treating all key events as global commands.

Claude-specific design:
- Claude-specific TUI is React/Ink-based. The central shape is `REPL.tsx` composing React components, hook state, `KeybindingSetup`, `AlternateScreen`, `FullscreenLayout`, `Messages`, and `PromptInput`.
- Claude-specific message display uses `normalizeMessages`, UI-only compact/snipped retention, grouping/collapse helpers, synthetic streaming tool-use messages with deterministic UUIDs, React memo comparison, and optional `VirtualMessageList`.
- Claude-specific streaming text preview shows only complete lines and is hidden for reduced motion or terminal bugs; cancel can promote partial preview text into a real assistant message.
- Claude-specific tool display delegates rendering to tool-owned JSX renderers through `AssistantToolUseMessage`.

Codex-specific design:
- Codex-specific TUI is Rust/Ratatui-based. The central shape is `App::run` multiplexing app/server/thread/terminal events and calling `ChatWidget::render` inside `Tui::draw_with_resize_reflow`.
- Codex-specific display unit is `HistoryCell`, with explicit `display_lines`, `raw_lines`, `display_hyperlink_lines`, `transcript_lines`, and wrapped height methods.
- Codex-specific live display separates committed `transcript_cells`, mutable `active_cell`, transcript overlay live-tail caching, pending history-line buffers, and bottom-pane renderables.
- Codex-specific bottom pane owns the composer and view stack, while `ChatWidget` owns higher-level interrupt/quit/task-running decisions.

Distilled harness rule:
- Build the TUI as a projection layer with named boundaries: event source, retained transcript/history state, active streaming state, renderable message/cell projection, input composer, modal/approval stack, keybinding precedence, frame scheduling, and debug/snapshot proof. For every TUI feature, mark whether it is display-only, runtime-control, model-input-producing, persisted, or eval/debug-only.

Anti-invention rule:
- Do not invent a generic "best" TUI architecture. If following Claude, cite the React/Ink REPL, `Messages`, `PromptInput`, virtual-scroll, and tool-owned JSX render paths. If following Codex, cite `Tui`, `App`, `ChatWidget`, `HistoryCell`, `BottomPane`, and Ratatui snapshot/test surfaces. If only one reference implements a mechanism, mark it source-specific.

Verification pattern:
- For a Claude-style harness, verify: streaming preview shows only complete lines and clears/promotes correctly on message/cancel; transcript mode chooses virtual scroll vs capped fallback exactly as configured; scroll keybindings precede cancel handling; synthetic streaming tool-use rows have stable keys; `Messages` transformations do not leak model-context filtering assumptions into display; PromptInput blocks/accepts submit according to footer, suggestions, images, speculation, and viewed-agent state; permission overlays remain visible and actionable while blocking runtime.
- For a Codex-style harness, verify: `TuiEvent` key/paste/resize/draw paths schedule frames and preserve terminal state; app loop handles app, thread, terminal, and app-server events without starving draw; committed `HistoryCell`s enter retained transcript plus terminal scrollback; active cells update live without committing prematurely; raw/rich/transcript display modes render distinct expected lines; bottom-pane cancellation is offered before process-level interrupt; status row animates/suppresses details deterministically; Ratatui snapshot tests cover narrow/wide/status/history cells.
