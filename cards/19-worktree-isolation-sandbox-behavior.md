Runtime trace evidence:
- Claude session worktrees are explicit runtime state. A slug is validated, flattened to `.claude/worktrees/<slug>`, a branch name is derived, then `git worktree add -B` or configured hooks create the isolated tree; creation can configure sparse checkout, copy `.worktreeinclude` matches, copy local settings, configure hooks, symlink configured directories, and record `currentWorktreeSession` plus project config state (`worktree.ts:48-87`, `worktree.ts:204-227`, `worktree.ts:229-375`, `worktree.ts:377-624`, `worktree.ts:702-778`).
- Claude startup `--worktree` moves from the invoked directory to the canonical main repo root if needed, creates the worktree, then `process.chdir(worktreePath)`, `setCwd(worktreePath)`, `setOriginalCwd(getCwd())`, `setProjectRoot(getCwd())`, saves state, clears CWD-dependent caches, and re-reads hooks/config from the worktree (`setup.ts:191-238`, `setup.ts:271-285`).
- Claude mid-session EnterWorktree is a model-visible tool but gated by its prompt: only when the user explicitly mentions worktree. It resolves to the main repo root, creates the worktree, switches process and shell CWD, saves worktree state, and clears prompt/memory/plans caches (`EnterWorktreeTool.ts:77-127`; `prompt.ts:1-30`).
- Claude ExitWorktree only operates on `getCurrentWorktreeSession()` from the current session. It refuses removal unless clean or `discard_changes` is true, restores original CWD/project state, clears caches, kills tmux on remove, and calls either `keepWorktree` or `cleanupWorktree` (`ExitWorktreeTool.ts:174-223`, `ExitWorktreeTool.ts:227-328`; `prompt.ts:1-31`).
- Claude agent isolation is separate from session isolation. The Agent tool schema has `isolation` and `cwd`; `cwd` is mutually exclusive with worktree isolation and overrides the working directory for all filesystem/shell operations. `createAgentWorktree` intentionally does not mutate global session state, process CWD, or project config (`AgentTool.tsx:90-101`; `worktree.ts:896-952`).
- Claude forked agents with worktree isolation receive an explicit notice that parent-context paths refer to the parent CWD, must be translated to the worktree root, and stale inherited files should be re-read (`AgentTool.tsx:595-640`; `forkSubagent.ts:200-210`).
- Claude agent worktree cleanup is result-dependent: hook-based worktrees are kept; git worktrees with no changes are removed and metadata is cleared; changed worktrees are kept and returned in sync/async result or notification metadata (`AgentTool.tsx:643-685`, `AgentTool.tsx:976-1035`, `AgentTool.tsx:1338-1372`; `runAgent.ts:732-742`).
- Claude Bash sandbox is config-driven. Settings declare filesystem allow/deny read/write paths, network allowed/denied domains, Unix socket and local-binding policy, `autoAllowBashIfSandboxed`, `allowUnsandboxedCommands`, `failIfUnavailable`, ignored violations, excluded commands, and platform-restriction settings (`sandboxTypes.ts:14-143`; `sandbox-adapter.ts:459-547`).
- Claude converts settings and permission rules into sandbox-runtime config. It always allows `.` and the Claude temp dir, denies writes to settings and `.claude/skills`, handles current-vs-original CWD, adds main repo write access when a git worktree is detected, includes additional directories, extracts WebFetch domain rules, and updates the runtime config on settings changes (`sandbox-adapter.ts:172-380`, `sandbox-adapter.ts:390-445`, `sandbox-adapter.ts:730-803`).
- Claude Bash decides whether to sandbox per command: sandbox must be enabled, `dangerouslyDisableSandbox` may bypass only if policy allows it, the command must exist, and excluded command patterns bypass sandboxing. The command execution path passes `shouldUseSandbox(input)` into `exec` (`shouldUseSandbox.ts:130-152`; `BashTool.tsx:881-898`).
- Codex has no source-grounded model-visible tool that creates a git worktree. Its worktree-specific evidence is config loading: linked worktree `.git` pointer tests keep worktree `.codex/config.toml` project layers while using root repo hooks (`config_loader_tests.rs:2261-2392`). Treat Claude worktree creation as source-specific.
- Codex resolves the effective `cwd` at config load from the override or current directory, normalizes it, resolves additional writable roots against it, discovers repo root for trust, and computes `workspace_roots`; defaults are `cwd` plus additional roots unless an explicit override replaces the list (`config/mod.rs:2320-2347`, `config/mod.rs:2822-2865`, `config/mod.rs:2910-2940`).
- Codex's canonical isolation unit is `PermissionProfile` plus `workspace_roots`. Built-in or named profiles compile to filesystem/network runtime policies; symbolic `:workspace_roots` entries are materialized against configured roots, then the effective profile is constrained by requirements and enriched with helper readable roots before being stored in `Config.permissions` (`config/mod.rs:2941-3078`, `config/mod.rs:3445-3545`; `permissions.rs:757-820`).
- Codex sandbox execution uses `SandboxManager::transform`. It sends command, cwd, permission profile, selected sandbox type, managed-network flag, sandbox-policy cwd, Linux helper path, and Windows settings into the transform; execution later either routes through a Windows sandbox backend or spawns the transformed command directly (`exec.rs:331-438`, `exec.rs:440-490`, `exec.rs:939-998`).
- Codex Linux sandbox is helper-based. The helper receives the canonical permission profile as JSON, derives runtime filesystem/network policy, defaults to bubblewrap plus seccomp according to the source comment, and runs the command with the command CWD and sandbox policy CWD separated (`landlock.rs:14-70`).
- Codex shell runtime preserves denied-read restrictions during escalation. If a command requests escalated permissions but the active filesystem policy has denied reads, `sandbox_permissions_preserving_denied_reads` downgrades back to `UseDefault` so denied reads remain enforced; managed network is dropped for escalated commands (`tools/sandboxing.rs:246-314`; `tools/runtimes/shell.rs:203-258`).

What the runtime does:
- Claude has two isolation layers: directory/worktree isolation changes the session or agent working tree, while Bash sandboxing wraps shell commands with filesystem/network restrictions. They interact because sandbox config includes current/original CWD, additional directories, and detected worktree main repo write access (`worktree.ts:702-778`; `sandbox-adapter.ts:222-299`, `sandbox-adapter.ts:282-288`).
- Claude CWD is runtime state, not just process state. `setCwd` resolves and realpaths a target before updating CWD state; shell execution can recover from deleted CWD by falling back to original CWD (`Shell.ts:447-465`, `Shell.ts:218-238`; `bootstrap/state.ts:500-532`).
- Claude file/path permission checks treat `getOriginalCwd()` plus additional working directories as the allowed working set, resolve symlinks and case, and reset shell CWD to original when it moves outside allowed paths or project-dir maintenance requires it (`filesystem.ts:667-744`; `BashTool/utils.ts:167-192`).
- Claude sandbox initialization is lazy but required before wrapping when sandboxing is enabled. It can fail soft with a warning, or hard-exit if `failIfUnavailable` is set; REPL initializes `SandboxManager` with a network ask callback and queues local/remote sandbox permission dialogs (`sandbox-adapter.ts:704-724`, `sandbox-adapter.ts:730-803`; `REPL.tsx:2216-2345`, `REPL.tsx:4594-4657`).
- Codex does not mutate process-wide cwd for isolation in the cited core path. It carries `cwd`, `workspace_roots`, `permission_profile`, and environment selections through config, turn context, shell requests, exec requests, and platform sandbox transforms (`config/mod.rs:3533-3545`; `tools/runtimes/shell.rs:54-72`; `exec.rs:331-438`).
- Codex filesystem enforcement is policy-driven. `FileSystemSandboxPolicy` resolves path access against CWD, materializes `:workspace_roots`, computes writable roots with read-only carveouts, and protects `.git`, `.agents`, and `.codex` metadata paths, including git pointer files used by worktrees/submodules (`permissions.rs:701-733`, `permissions.rs:757-820`, `permissions.rs:989-1103`, `permissions.rs:1593-1630`).
- Codex platform selection happens below the harness policy: `SandboxPolicy`/`PermissionProfile` express the intended disk/network access, then sandboxing backends such as Linux helper or Windows restricted/elevated backends enforce or reject if they cannot safely enforce the policy (`exec.rs:331-438`, `exec.rs:1000-1078`; `landlock.rs:14-70`).

Shared invariant:
- Isolation is a multi-plane contract. A harness must keep model-visible isolation hints, runtime CWD/workspace-root selection, sandbox policy/enforcement, persisted session/turn state, user-visible status, and debug/eval evidence separate. Do not infer a worktree mechanism from a sandbox policy, or a sandbox policy from a worktree path.

Claude-specific design:
- Claude has first-class worktree creation/removal tools and startup support. It can mutate session CWD into `.claude/worktrees/...`, preserve or remove the worktree, and store current worktree state. This is source-specific; Codex does not show an equivalent model-visible worktree creation tool.
- Claude also has Agent-tool worktree isolation that avoids mutating global session state. Agent worktrees are cleaned up automatically only when source evidence shows no changes; changed or hook-based worktrees are kept and surfaced to the caller.
- Claude Bash sandbox is an external `@anthropic-ai/sandbox-runtime` adapter whose config is derived from settings and permission rules, updated on settings changes, and visible to the model through Bash prompt restrictions.
- Claude treats CWD as a mutable shell/session state but guards it with allowed-working-directory checks, CWD reset warnings, and cache invalidation when entering/leaving worktrees.

Codex-specific design:
- Codex's primary isolation mechanism is not worktree creation; it is a canonical permission profile plus runtime workspace roots, carried through config, turn context, model-visible environment context, shell runtime, and exec sandbox transform.
- Codex preserves symbolic `:workspace_roots` in durable profiles and materializes them against effective runtime roots for model context and execution. This lets cwd changes avoid silently rebinding durable project-root authority unless the source path explicitly updates workspace roots.
- Codex separates command CWD from sandbox-policy CWD and platform backend details. Linux uses a helper executable with the permission profile; Windows backends either enforce policy or reject unsafe fallback.
- Codex protects project metadata paths under writable roots, including `.git` pointer files used by worktrees/submodules, but this is metadata protection inside filesystem sandboxing, not a worktree lifecycle feature.

Distilled harness rule:
- Build isolation as explicit, typed state that crosses every harness boundary: model-visible prompt/schema, runtime CWD/workspace roots, filesystem/network sandbox policy, command execution wrapper, persisted session/turn metadata, user display, and tests. If a source creates worktrees, cite the creation/removal path; if it only constrains workspace roots, cite that instead.

Anti-invention rule:
- Do not collapse "worktree", "workspace root", "cwd", "sandbox", and "permission profile" into one generic isolation feature. Claude worktree creation is Claude-specific. Codex workspace-root permission profiles are Codex-specific. A shared rule may only say that both keep isolation state explicit and separately projected across model/runtime/persistence/display/eval planes.

Verification pattern:
- For any harness isolation mechanism, verify in order: model-visible control or environment hint, runtime CWD/workspace-root derivation, filesystem/network policy construction, command execution wrapper/backend, cleanup/exit behavior, persisted session/turn/config state, user-visible status or warning, debug logs, and tests. For worktrees specifically, additionally verify slug/path safety, original CWD, branch/head baseline, change detection before removal, cache invalidation, and whether global session state is mutated.

Skill text candidate:
- When studying a coding-agent harness's isolation behavior, split the evidence into two questions: "Where does work happen?" and "What can code running there read/write/connect to?" The first question is answered by CWD, worktree, workspace-root, and environment-selection sources. The second is answered by permission-profile, filesystem/network sandbox, and exec-backend sources. Only after both are cited should you produce a reusable rule, and source-specific mechanisms must remain labeled as such.
