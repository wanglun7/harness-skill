Runtime trace evidence:
- Claude wraps provider calls in `withRetry`. The loop checks abort before each attempt, refreshes clients or auth caches on stale/auth errors, drops background 529 retries, treats persistent 429/529 as an unattended keep-alive mode, obeys retry headers/backoff, yields retry `SystemAPIErrorMessage` records, and finally throws `CannotRetryError` when retry gates fail (`withRetry.ts:170-260`, `withRetry.ts:316-516`, `withRetry.ts:696-760`).
- Claude has source-specific fast-mode and overload behavior. A 429/529 can disable fast mode, enter cooldown, or after consecutive foreground 529s throw `FallbackTriggeredError` to switch models; non-foreground 529s are explicitly not amplified (`withRetry.ts:57-109`, `withRetry.ts:261-365`).
- Claude recovers from some API errors by changing future request shape, not by surfacing the first error. A max-token/context-overflow 400 can reduce `maxTokensOverride`; prompt-too-long or media-size failures can try context-collapse drain or reactive compaction before yielding the withheld error (`withRetry.ts:384-427`, `query.ts:1062-1175`).
- Claude's query loop treats `FallbackTriggeredError` as a structured transition: it fills missing tool results, discards the streaming tool executor, switches the main model, strips thinking signatures for a source-specific user type, emits a warning system message, and continues (`query.ts:900-951`).
- Claude abort handling is transcript-shape aware. On user abort during streaming it consumes remaining tool results when available, synthesizes `Interrupted by user` tool results for missing tool ids when needed, cleans up computer-use state, optionally emits a user interruption, and returns `aborted_streaming` (`query.ts:1011-1052`).
- Claude separates API error translation from retry. `getAssistantMessageFromError` maps timeout, rate-limit, prompt-too-long, PDF/image/request-size, tool-use corruption, duplicate tool ids, invalid model, billing, auth, 404, and connection errors into assistant API-error messages with selected `error` categories and sometimes raw details (`errors.ts:425-934`).
- Claude generic error taxonomy is local and pragmatic: `AbortError`, SDK `APIUserAbortError`, `ShellError`, telemetry-safe errors, errno extraction, filesystem-inaccessible buckets, and axios auth/timeout/network/http/other buckets (`utils/errors.ts:1-238`).
- Claude tool failures become model-visible `tool_result` errors. Zod input failures are tagged as `<tool_use_error>InputValidationError...`; runtime exceptions are formatted by `formatError`, logged unless abort-shaped, run post-tool-use-failure hooks, and return `is_error: true` with `toolUseResult: Error: ...` (`toolExecution.ts:599-680`, `toolExecution.ts:1589-1737`).
- Claude streaming tool execution creates synthetic error messages for already-aborted tools, bubbles non-sibling aborts to the query abort controller, and cancels sibling tools only when a Bash tool produces an error result (`StreamingToolExecutor.ts:254-365`).
- Claude max-output recovery is a model-visible continuation strategy: before surfacing the final error, it adds a meta user message telling the model to resume directly and increments a recovery counter; API-error final messages skip normal stop hooks and run stop-failure hooks to avoid an error/hook retry loop (`query.ts:1223-1265`).
- Codex has a central `CodexErr` taxonomy with explicit retryability. Non-retryable categories include turn abort, interruption, invalid request/image, usage/quota limits, server overloaded, cyber policy, sandbox, unsupported operations, fatal, and context-window exceeded; retryable categories include stream, timeout, request timeout, unexpected status, response-stream/connection failure, internal server error, internal agent died, IO/JSON/join errors (`error.rs:67-210`).
- Codex maps API/SSE failures into that taxonomy before turn handling. `response.failed` classifies context window, quota, usage-not-included, cyber policy, invalid prompt, overloaded, or retryable-with-delay; the bridge maps HTTP 503 overloaded, 400 cyber/invalid image/invalid request, 429 usage limits versus retry limit, 500 internal, timeout, network/build, and rate-limit errors (`responses.rs:347-375`, `responses.rs:531-570`, `api_bridge.rs:18-137`).
- Codex turn retry is localized in `run_sampling_request`. It preserves the original request input across retries, updates token/full-rate-limit state for non-retryable context/usage errors, checks `CodexErr::is_retryable`, calls `handle_retryable_response_stream_error`, and records a sampling retry (`turn.rs:1088-1140`).
- Codex retry recovery includes UI notification and transport fallback. Retryable stream errors use requested delay or backoff, send `StreamError` notifications such as reconnecting messages after the first hidden websocket retry, and after retry budget can switch a session from WebSocket to HTTPS transport with a warning (`responses_retry.rs:20-79`, `client.rs:1672-1688`).
- Codex auth recovery is client-level. HTTP 401 in HTTP or WebSocket setup calls `handle_unauthorized`, marks the next attempt with `PendingUnauthorizedRetry`, retries the API call, and records auth retry/recovery metadata in API telemetry and feedback tags (`client.rs:1263-1364`, `client.rs:1393-1465`, `client.rs:1912-1970`, `client.rs:2138-2185`).
- Codex turn errors remain recoverable at the conversation level unless they abort the turn. Invalid-image errors first sanitize last-turn images and retry; otherwise terminal errors emit turn-error lifecycle, analytics, and `ErrorEvent`, then the user can continue the conversation (`turn.rs:372-409`).
- Codex tool errors are split between model-handled and fatal. `FunctionCallError::RespondToModel` becomes a function/custom-tool output with `success: false`; `FunctionCallError::Fatal` becomes `CodexErr::Fatal` and terminates the turn. Invalid JSON, blocked pre/post hooks, missing tool output, nonzero exec exits, sandbox denial/timeout, and user rejection each have source-specific projection paths (`function_call_error.rs:3-10`, `parallel.rs:62-79`, `parallel.rs:186-220`, `registry.rs:445-510`, `registry.rs:620-650`, `events.rs:344-430`).
- Codex terminal errors are persisted and surfaced through event machinery. `send_event` records error events to rollout/thread traces and stores terminal-error text when `CodexErrorInfo` affects turn status; stream errors are separate intermediate events with additional details (`session/mod.rs:1645-1678`, `session/mod.rs:3386-3402`).

What the model sees:
- Claude's model sees assistant API-error messages when API failures are surfaced as conversation messages, model-visible retry/meta continuation messages for max-output recovery, and `tool_result` blocks with `is_error: true` for tool validation/runtime errors (`errors.ts:425-934`, `query.ts:1223-1237`, `toolExecution.ts:664-680`, `toolExecution.ts:1715-1737`).
- Claude's model can see synthetic tool results produced by abort/fallback cleanup so provider history preserves `tool_use`/`tool_result` pairing after interruption or model fallback (`query.ts:900-951`, `query.ts:1011-1052`, `StreamingToolExecutor.ts:277-291`).
- Claude's model does not see raw retry loop internals such as backoff, client refresh, 529 counters, or debug logs unless those paths yield a system retry message or final assistant API-error message (`withRetry.ts:465-512`, `errors.ts:425-934`).
- Codex's model sees successful retry attempts as ordinary prompt history; intermediate retry `StreamError` notifications are client events, not model items. Tool-level nonfatal failures are returned as `FunctionCallOutput` or custom-tool output with `success: false` (`responses_retry.rs:64-75`, `parallel.rs:186-220`).
- Codex's model may see sanitized history after invalid-image recovery because `replace_last_turn_images("Invalid image")` mutates history before retrying. Terminal `ErrorEvent` messages are user/client events, not function-call outputs (`turn.rs:376-409`).

What the runtime does:
- Claude runtime classifies, retries, and sometimes mutates state before surfacing errors. It may clear AWS/GCP auth caches, refresh clients, disable fast mode, adjust max output tokens, switch fallback model, compact, drain context-collapse output, or synthesize missing tool results (`withRetry.ts:170-516`, `query.ts:900-1175`).
- Claude runtime avoids recursive failure loops by withholding recoverable API errors until recovery is exhausted and by skipping normal stop hooks when the last message is an API error (`query.ts:1062-1175`, `query.ts:1254-1265`).
- Claude runtime records tool failure spans/events, marks MCP auth clients as `needs-auth`, and uses abort-shaped errors to distinguish interruption from ordinary tool failure (`toolExecution.ts:1593-1694`).
- Codex runtime first maps provider/transport failures into `CodexErr`, then lets turn code decide retryability, fallback transport, terminal event emission, analytics tracking, and continuation eligibility (`api_bridge.rs:18-137`, `error.rs:172-259`, `turn.rs:372-409`, `turn.rs:1088-1140`).
- Codex runtime keeps `ModelClientSession` turn-scoped so WebSocket/sticky-routing state survives retries within a turn; fallback transport changes that session state for subsequent attempts (`turn.rs:200-206`, `responses_retry.rs:31-46`, `client.rs:1672-1688`).
- Codex runtime routes recoverable stream failures through `StreamError` notifications while terminal errors go through `ErrorEvent` and turn status failure; app-server keeps those paths separate (`session/mod.rs:3386-3402`, `bespoke_event_handling.rs:920-972`).

What gets persisted:
- Claude persists error-shaped assistant/user/tool-result messages when they are yielded into the conversation transcript. Tool errors include model-visible `tool_result` content and `toolUseResult`; API retry internals and debug logs are not themselves the transcript source of truth unless emitted as messages (`toolExecution.ts:664-680`, `toolExecution.ts:1715-1737`, `query.ts:1223-1237`).
- Claude query engine also accumulates turn-scoped in-memory errors for SDK result objects; a final non-successful result can include diagnostic text and errors after the watermark (`QueryEngine.ts:1082-1117`).
- Codex persists `EventMsg::Error` and `EventMsg::StreamError` through rollout/thread trace recording; terminal errors can also set `turn_context.terminal_error`, and analytics stores turn error kind plus HTTP status code (`session/mod.rs:1645-1678`, `analytics/src/facts.rs:122-236`).
- Codex tool failures that are returned to the model persist as response input items/function outputs in the conversation history; fatal turn errors persist as event messages rather than model function outputs (`parallel.rs:186-220`, `session/mod.rs:1645-1678`).

What the user/TUI sees:

Shared invariant:
- A top-tier harness separates provider/transport errors, runtime/tool errors, user interruption, policy/sandbox rejection, model-visible error payloads, terminal user events, persisted transcript/event state, and telemetry/eval labels.
- Retryability is a typed runtime decision, not a generic catch-and-retry habit. Recovery must preserve conversation invariants such as tool-call pairing, original request input, turn-scoped client/session state, and user-visible liveness.
- Nonfatal tool errors should usually be returned to the model as structured failed outputs; fatal harness errors should end or abort the turn through a terminal event path.

Claude-specific design:
- Claude uses `withRetry` plus `CannotRetryError` and `FallbackTriggeredError` rather than a single central error enum. Its recovery paths are embedded in API retry and `query.ts` state transitions.
- Claude has source-specific fast-mode disabling/cooldown, foreground-only 529 retry/fallback model switching, persistent unattended 429/529 keep-alive messages, reactive compaction after prompt/media errors, and max-output meta continuation.
- Claude tool errors use Anthropic-style `tool_result` blocks with `is_error: true`, `<tool_use_error>` tags for validation, and display normalization in React/TUI components.

Codex-specific design:
- Codex centralizes error taxonomy in `CodexErr`, maps API/SSE/transport errors into it before turn handling, and exposes retryability through `CodexErr::is_retryable`.
- Codex response retry can switch WebSocket sessions to HTTPS transport after exhausting retry budget, while preserving a turn-scoped `ModelClientSession`.
- Codex separates `StreamError` intermediate retry notifications from terminal `ErrorEvent`s, app-server `TurnError`s, and TUI finalization paths.
- Codex tool failure shape is Responses-oriented: `FunctionCallError::RespondToModel` becomes `FunctionCallOutput` or custom-tool output with `success: false`, while `Fatal` becomes `CodexErr::Fatal`.

Distilled harness rule:
- Implement errors as a source-typed recovery matrix: category, retryability, retry delay, state mutation before retry, model-visible projection, terminal user event, persisted record, and telemetry/eval label. A retry path is complete only if it preserves prompt/tool invariants and gives the user a liveness signal when the wait is nontrivial.

Anti-invention rule:
- Do not invent error classes, retry budgets, fallback triggers, compaction triggers, transport fallback rules, or tool-error display conventions. If following Claude, cite the exact `withRetry`, `query.ts`, API-error mapper, and tool-result paths. If following Codex, cite `CodexErr`, API/SSE bridge mapping, `run_sampling_request`, `responses_retry`, and tool `FunctionCallError` paths. If only one source implements a recovery behavior, mark it source-specific.

Verification pattern:
- For a Claude-style harness, verify: abort before request throws user-abort; non-foreground 529 is dropped; persistent 429/529 yields keep-alive retry messages; fast-mode 429/529 disables or cools down fast mode; repeated foreground 529 triggers fallback transition and synthetic tool results; max-token overflow adjusts `maxTokensOverride`; prompt-too-long/media errors attempt compact/recovery once; max-output recovery injects a meta continuation and stops after the configured limit; tool validation/runtime errors return `is_error` tool results and display cleanly.
- For a Codex-style harness, verify: each API/SSE/transport failure maps to the expected `CodexErr`; `is_retryable` blocks terminal classes and permits only source-defined transient classes; retryable streams emit reconnect notifications and sleep with requested delay/backoff; WebSocket fallback switches to HTTPS after retry budget; 401 recovery retries once with auth telemetry; invalid-image history is sanitized and retried; terminal errors emit lifecycle, analytics, rollout event, app-server notification, and TUI failed-state handling.
